## Finding Details

### #1 — Node.js Dependency Vulnerabilities (HIGH)

**Location:** All 3 package-lock.json files
**Category:** A06 — Vulnerable and Outdated Components

Multiple HIGH-severity CVEs in transitive dependencies:
- **hono** ≤4.12.6 — 5 advisories: timing attack in basicAuth, cookie injection, SSE injection, prototype pollution, arbitrary file access
- **path-to-regexp** 8.0.0-8.3.0 — ReDoS via sequential optional groups
- **express-rate-limit** 8.2.0-8.2.1 — IPv4-mapped IPv6 bypass

**Exploit Scenario:** Attacker crafts request with malicious cookie domain attribute → cookie injection → session hijacking (if hono routes are exposed).

**Remediation:** Run `npm audit fix` across all package directories. Estimated time: 30 minutes.

---

### #2 — Process Engine Webhooks Lack Signature (HIGH)

**Location:** `src/backend/services/process_engine/events/webhook_publisher.py:100-117`
**Category:** A08 — Software and Data Integrity Failures

Outbound webhook calls to external systems include no HMAC signature header. Recipients cannot verify Trinity as the origin.

**Exploit Scenario:** Attacker sends forged POST to a webhook receiver URL with a fake `process_completed` event → triggers downstream automation (CI deploy, Slack alert, database update) without Trinity ever running the process.

**Remediation:** Add `X-Trinity-Signature` HMAC-SHA256 header using a per-webhook secret. ~2-4 hours.

---

### #3 — Credential Encryption Key as Env Var (HIGH)

**Location:** docker-compose.yml, .env
**Category:** A02 — Cryptographic Failures

The AES-256-GCM encryption key for all `.credentials.enc` files is a plaintext environment variable. Visible in `docker inspect`, process listings, and container environment. No rotation mechanism exists.

**Exploit Scenario:** Attacker with read access to any backend container process listing extracts CREDENTIAL_ENCRYPTION_KEY → decrypts all agent credential backups.

**Remediation:** Migrate to Docker secrets (Swarm mode) or external secret manager. Implement key rotation policy. Priority: before production deployment.

---

### #4 — Missing CODEOWNERS (MEDIUM)

**Location:** `.github/CODEOWNERS` — absent
**Category:** CI/CD Process

No mandatory code review enforcement for security-sensitive files (auth routes, Docker configs, workflows).

**Remediation:** Create CODEOWNERS requiring review for `src/backend/routers/auth.py`, `src/backend/routers/internal.py`, `.github/workflows/`, `docker-compose*.yml`. Time: 15 minutes.

---

### #5 — Missing USER Directive in Dockerfiles (MEDIUM)

**Location:** `docker/backend/Dockerfile`, `docker/frontend/Dockerfile`, `docker/frontend/Dockerfile.prod`
**Category:** Container Security

All three containers run as root. If a process is compromised, the attacker has root privileges within the container.

**Note:** Docker security options (cap_drop ALL, no-new-privileges) are in place, which significantly reduces impact.

**Remediation:** Add `USER` directives to run as non-root. ~1 hour per Dockerfile.

---

### #6 — JWT 7-Day Lifetime (MEDIUM)

**Location:** `src/backend/config.py:26`
**Category:** A07 — Identification and Authentication Failures

`ACCESS_TOKEN_EXPIRE_MINUTES = 10080` (7 days) with no refresh token mechanism. If a token is compromised, the attacker has a 7-day window.

**Remediation:** Reduce to 24 hours, implement refresh tokens, or add token revocation. Priority: medium-term.

---

### #7 — Scoped Secret Exposure in CI (MEDIUM)

**Location:** `.github/workflows/publish-cli.yml:88`

`GH_TOKEN: ${{ secrets.HOMEBREW_TAP_TOKEN }}` set at job level — visible to all steps in the job.

**Remediation:** Move to step-level `env:` block. Time: 5 minutes.

---

### #8 — No Python Lockfiles (MEDIUM)

**Location:** `docker/scheduler/requirements.txt`, `tests/requirements-test.txt`

No `poetry.lock`, `Pipfile.lock`, or `pip-compile` output tracked in git. Transitive dependency versions are unpinned.

**Remediation:** Add `pip-compile` to build process or migrate to Poetry. Time: 1-2 hours.
