# Operator Queue & Operating Room

> **Requirement ID**: OPS-001
> **Priority**: HIGH
> **Status**: In Progress (Phase 3 UI mockup complete)
> **Date**: 2026-03-07

---

## Vision

A unified "Operating Room" interface where a human operator monitors, controls, and collaborates with Trinity agents in real time. Agents communicate with the operator through a standardized file-based protocol — writing requests to a JSON queue file in their workspace, which the platform syncs to a structured database and presents in a single actionable UI.

The system has two sides:
1. **Agent-side**: Default system prompt instructions that inform agents about the operator queue protocol. Agents are free to use it or not — it's an available mechanism, not a mandatory behavior.
2. **Platform-side**: Polling/sync infrastructure, database persistence, and the Operating Room UI

## 1. Agent-Side Protocol

### 1.1 The Operator Queue File

Each agent maintains a file at `~/.trinity/operator-queue.json` in their workspace. This file is the shared mailbox between the agent and the platform.

**Location**: `/home/developer/.trinity/operator-queue.json`

**Schema**:

```json
{
  "$schema": "operator-queue-v1",
  "requests": [
    {
      "id": "req-<timestamp>-<sequential>",
      "type": "approval | question | alert",
      "status": "pending | responded | acknowledged",
      "priority": "critical | high | medium | low",
      "title": "Short summary (displayed in queue list)",
      "question": "Full description of what the agent needs",
      "options": ["approve", "reject", "defer"],
      "context": {
        "execution_id": "exec-...",
        "task": "What the agent was doing when it raised this",
        "relevant_data": {}
      },
      "created_at": "2026-03-07T10:00:00Z",
      "expires_at": "2026-03-07T11:00:00Z",
      "response": null,
      "response_text": null,
      "responded_by": null,
      "responded_at": null
    }
  ]
}
```

**Field Definitions**:

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | string | Yes | Unique ID generated by agent. Format: `req-<YYYYMMDD>-<NNN>` |
| `type` | enum | Yes | `approval` (yes/no decision), `question` (freeform answer), `alert` (acknowledgement only) |
| `status` | enum | Yes | `pending` (awaiting response), `responded` (operator answered), `acknowledged` (agent processed the response) |
| `priority` | enum | Yes | `critical` (needs immediate attention), `high`, `medium`, `low` |
| `title` | string | Yes | Short summary, max 120 chars. Displayed as queue item title. |
| `question` | string | Yes | Full description. Can include markdown. |
| `options` | string[] | For approvals | Predefined choices. For `approval` type, at least 2 options. Ignored for `question` and `alert` types. |
| `context` | object | No | Structured metadata. Should include `execution_id` if available, plus any relevant data the operator needs to make a decision. |
| `created_at` | ISO 8601 | Yes | When the agent created this request. |
| `expires_at` | ISO 8601 | No | Optional deadline. Platform marks as expired if no response by this time. |
| `response` | string | No | For `approval`: the selected option. For `question`: freeform text. For `alert`: `"acknowledged"`. Written by platform. |
| `response_text` | string | No | Optional additional notes from the operator (for approvals that need explanation). Written by platform. |
| `responded_by` | string | No | Email of the operator who responded. Written by platform. |
| `responded_at` | ISO 8601 | No | When the operator responded. Written by platform. |

### 1.2 Request Types

**Approval** — Agent needs a yes/no or multi-choice decision before proceeding.
```json
{
  "id": "req-20260307-001",
  "type": "approval",
  "status": "pending",
  "priority": "high",
  "title": "Deploy PR #47 to staging?",
  "question": "PR has 47 changed files across 3 services. All tests pass. Ready to deploy to staging environment.",
  "options": ["approve", "reject", "defer to next week"],
  "context": {
    "execution_id": "exec-abc123",
    "task": "deploy-pipeline",
    "pr_url": "https://github.com/org/repo/pull/47",
    "test_results": "142/142 passed",
    "files_changed": 47
  },
  "created_at": "2026-03-07T10:00:00Z",
  "expires_at": "2026-03-07T12:00:00Z"
}
```

**Question** — Agent needs freeform input or guidance.
```json
{
  "id": "req-20260307-002",
  "type": "question",
  "status": "pending",
  "priority": "medium",
  "title": "Which database migration strategy?",
  "question": "The schema change can be applied as a single migration (faster, brief downtime) or as a multi-step zero-downtime migration (slower, no downtime). Which approach should I use?",
  "options": null,
  "context": {
    "execution_id": "exec-def456",
    "task": "schema-migration",
    "migration_file": "migrations/0024_add_operator_queue.sql",
    "estimated_downtime_single": "~30 seconds",
    "estimated_time_zero_downtime": "~15 minutes"
  },
  "created_at": "2026-03-07T10:15:00Z"
}
```

**Alert** — Agent is reporting a situation. No decision needed, just acknowledgement.
```json
{
  "id": "req-20260307-003",
  "type": "alert",
  "status": "pending",
  "priority": "critical",
  "title": "API rate limit exceeded for OpenAI",
  "question": "Hit rate limit on OpenAI API (429 Too Many Requests). Pausing all tasks that require OpenAI for 15 minutes. Will retry automatically.",
  "options": null,
  "context": {
    "execution_id": "exec-ghi789",
    "task": "content-generation",
    "error_code": 429,
    "retry_at": "2026-03-07T10:30:00Z"
  },
  "created_at": "2026-03-07T10:15:00Z"
}
```

### 1.3 File Lifecycle

1. **Active items only** — The file should contain only `pending` and `responded` (not yet processed by agent) items, plus up to 3 most recent `acknowledged` items for immediate context.
2. **Agent writes** `pending` entries when it needs something.
3. **Platform writes** `responded` status with `response`, `responded_by`, `responded_at` fields.
4. **Agent reads** responses, acts on them, then updates status to `acknowledged`.
5. **Agent cleans up** — removes `acknowledged` items older than the 3 most recent. The DB is the permanent record.

### 1.4 Default System Prompt Integration

The operator queue protocol is communicated to agents through the **Trinity meta-prompt** (`config/trinity-meta-prompt/prompt.md`) — the default system prompt injected into every Trinity agent at deployment.

Agents are informed about the mechanism and the file protocol. They are **free to use it or not** — it's an available communication channel, not a mandatory behavior. The agent decides when and whether to ask for human input based on its own judgment and context.

**Integration point**: Add an "Operator Communication" section to the existing meta-prompt.

**What the meta-prompt tells agents:**

1. There is a file at `~/.trinity/operator-queue.json` for communicating with the human operator
2. The JSON schema and the three request types (approval, question, alert)
3. How to read operator responses (check for `status: "responded"` items)
4. That they should update status to `"acknowledged"` after processing a response
5. That they can learn from responses and update their own memory if a response contains a reusable insight

**What the meta-prompt does NOT prescribe:**
- When to use it — the agent decides
- How often to check — the agent decides
- Whether to escalate — the agent decides
- Specific rules about what requires approval vs. what doesn't — that's up to the agent's own instructions or judgment

This keeps the protocol lightweight. Individual agent templates can add more specific guidance in their own CLAUDE.md if needed (e.g., "always get approval before deploying").

## 2. Platform-Side Implementation

### 2.1 Database Schema

**Table: `operator_queue`**

```sql
CREATE TABLE operator_queue (
    id TEXT PRIMARY KEY,                    -- Same ID as in JSON file (req-YYYYMMDD-NNN)
    agent_name TEXT NOT NULL,               -- Which agent created this
    type TEXT NOT NULL,                     -- approval, question, alert
    status TEXT NOT NULL DEFAULT 'pending', -- pending, responded, acknowledged, expired, cancelled
    priority TEXT NOT NULL DEFAULT 'medium',-- critical, high, medium, low
    title TEXT NOT NULL,                    -- Short summary
    question TEXT NOT NULL,                 -- Full description
    options TEXT,                           -- JSON array of options (for approvals)
    context TEXT,                           -- JSON object with structured metadata
    execution_id TEXT,                      -- FK to schedule_executions if available
    created_at TEXT NOT NULL,               -- ISO 8601
    expires_at TEXT,                        -- Optional deadline
    response TEXT,                          -- Selected option or freeform text
    response_text TEXT,                     -- Additional notes from operator
    responded_by_id TEXT,                   -- FK to users table
    responded_by_email TEXT,               -- Denormalized for display
    responded_at TEXT,                      -- ISO 8601
    acknowledged_at TEXT,                   -- When agent processed the response
    FOREIGN KEY (responded_by_id) REFERENCES users(id)
);

CREATE INDEX idx_operator_queue_agent ON operator_queue(agent_name);
CREATE INDEX idx_operator_queue_status ON operator_queue(status);
CREATE INDEX idx_operator_queue_priority ON operator_queue(priority);
CREATE INDEX idx_operator_queue_type ON operator_queue(type);
CREATE INDEX idx_operator_queue_created ON operator_queue(created_at DESC);
```

### 2.2 File Sync Service

**Location**: `src/backend/services/operator_queue_service.py`

A background service that syncs operator queue files with the database.

**Responsibilities**:
1. Poll agent operator-queue.json files on a configurable interval (default: 5 seconds)
2. Detect new `pending` entries → create DB records, broadcast WebSocket event
3. Detect `acknowledged` entries → update DB records
4. Write operator responses back to agent JSON files
5. Handle expired entries (check `expires_at`, update status)

**Polling Strategy**:
- Only poll running agents
- Compare file mtime to avoid unnecessary reads
- On each poll cycle:
  1. Read JSON file from agent container via existing file API (`GET /api/agents/{name}/files/download?path=.trinity/operator-queue.json`)
  2. Diff against known state in DB
  3. Sync changes in both directions
  4. Write back to agent if platform has new responses

**Alternative to Polling** (future optimization):
- Claude Code hooks could trigger on file write to `~/.trinity/operator-queue.json`
- Agent-server could expose a webhook endpoint that Claude Code hooks call
- This would replace polling with event-driven sync

### 2.3 Backend Router

**Location**: `src/backend/routers/operator_queue.py`

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| GET | `/api/operator-queue` | JWT | List all queue items with filters |
| GET | `/api/operator-queue/{id}` | JWT | Get single queue item |
| POST | `/api/operator-queue/{id}/respond` | JWT | Submit response to a queue item |
| POST | `/api/operator-queue/{id}/cancel` | JWT | Cancel a pending item |
| GET | `/api/operator-queue/stats` | JWT | Counts by status, type, priority, agent |
| GET | `/api/operator-queue/agents/{name}` | JWT | Queue items for specific agent |

**Query Parameters for List**:
- `status` — filter by status (pending, responded, acknowledged, expired)
- `type` — filter by type (approval, question, alert)
- `priority` — filter by priority
- `agent_name` — filter by agent
- `since` — items created after this timestamp
- `limit` / `offset` — pagination

**Respond Endpoint Body**:
```json
{
  "response": "approve",
  "response_text": "Looks good, go ahead with the deploy."
}
```

## 6. Integration Points with Existing Systems

### Notifications (NOTIF-001)
The operator queue **supersedes** the notification system for actionable items. Existing notifications become `alert` type queue entries. The Events page can link to the Operating Room for items that need response.

### Process Engine Approvals
Process Engine human approvals (`process_approvals` table) are a separate, process-scoped mechanism. The Operating Room should **also display** pending process approvals in a unified view. This can be achieved by querying both `operator_queue` and `process_approvals` tables and merging results in the UI.

### Agent Monitoring (MON-001)
Health alerts from the monitoring service can create `alert` type entries in the operator queue, making them visible in the Operating Room alongside agent-initiated requests.

### Execution Origin Tracking (AUDIT-001)
Queue items should link to execution records via `execution_id` in the context field, enabling the operator to trace from a queue item to the full execution that generated it.

## 7. Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Communication mechanism | File-based JSON | In-context for agents, survives restarts, no blocking, simple |
| Sync direction | Bidirectional (agent writes requests, platform writes responses) | Both sides need to write |
| Polling vs events | Polling initially, hooks as optimization | Polling is simpler to implement; hooks can replace later |
| History storage | DB only (file has active items + 3 recent) | Keeps agent context clean, DB is system of record |
| Memory integration | Agent-driven, self-directed | Agents know their own memory mechanisms; meta-prompt describes the option, agents decide |
| Queue item IDs | Agent-generated (`req-YYYYMMDD-NNN`) | Agent creates the item, ID should be deterministic from agent side |
| Response channel | Platform writes to agent's JSON file | Symmetric to agent writes; no new infrastructure needed |

## 8. Non-Functional Requirements

| Requirement | Target |
|-------------|--------|
| Sync latency (agent writes → UI shows) | < 10 seconds |
| Response delivery (operator responds → agent sees) | < 10 seconds |
| Concurrent agents polled | 50+ |
| Queue items displayed | Paginated, 50 per page |
| WebSocket update latency | < 500ms |
| File write safety | Atomic write (write temp file, rename) to prevent corruption |

## 9. Open Questions

1. **Should the system agent auto-process certain queue items?** — e.g., auto-acknowledge low-priority alerts, auto-approve items matching certain patterns. This could be a Phase 5+ feature where operators define rules.

2. **Multi-operator support** — Should queue items be assignable to specific operators? Or is first-come-first-serve sufficient for v1?

3. **Queue item threading** — Should agents be able to follow up on a responded item (e.g., "you said X, but I found Y — should I still proceed?")? This could be modeled as a new queue item with a `parent_id` reference.

4. **Batch operations** — Should the Operating Room support bulk acknowledge/respond for similar items?
